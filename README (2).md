# The Press Room

A full-stack news portal platform built with TypeScript, Express, React, and PostgreSQL. Readers subscribe to publishers and journalists, journalists write articles and newsletters, and editors approve content before it goes live.

---

## Features

- **Role-based access control** — three distinct roles (Reader, Journalist, Editor) with server-enforced permissions
- **Article approval workflow** — all articles start pending; editors review and approve before they appear publicly
- **Newsletters** — journalists curate collections of articles into themed newsletters
- **Publishers** — media organisations that journalists can post under
- **Personalised feed** — readers follow publishers and individual journalists and get a combined article feed
- **JWT authentication** — stateless auth with 7-day tokens stored in localStorage
- **Contract-first API** — OpenAPI spec drives Zod validation schemas and React Query hooks (generated, never hand-written)

## Getting Started

### Prerequisites

- Node.js 24+
- pnpm 9+
- PostgreSQL 15+

### 1. Install dependencies

```bash
pnpm install
```

### 2. Configure environment variables

Create a `.env` file in the project root:

```env
DATABASE_URL=postgresql://user:password@localhost:5432/pressroom
SESSION_SECRET=your-very-long-random-secret-here
```

Generate a secure secret with:
```bash
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
```

### 3. Set up the database

```bash
pnpm --filter @workspace/db run push
```

### 4. Start the servers

```bash
# Terminal 1 — API server (port 8080)
pnpm --filter @workspace/api-server run dev

# Terminal 2 — Frontend (port 21440)
pnpm --filter @workspace/news-portal run dev
```

The app is available at `http://localhost/`.

## Roles & Permissions

| Action | Reader | Journalist | Editor |
|--------|:------:|:----------:|:------:|
| Browse articles & newsletters | Yes | Yes | Yes |
| Register / log in | Yes | Yes | Yes |
| Follow publishers & journalists | Yes | — | — |
| View personalised feed | Yes | — | — |
| Write and submit articles | — | Yes | Yes |
| Create newsletters | — | Yes | Yes |
| Edit own articles | — | Yes | Yes |
| View pending articles queue | — | — | Yes |
| Approve or reject articles | — | — | Yes |

Roles are assigned at registration and enforced server-side on every protected route via the `requireRole` middleware. A journalist submitting an article cannot approve it — only an editor can move it from `pending` to `approved`.

## API Overview

All endpoints are prefixed with `/api`. Authentication uses `Authorization: Bearer <token>`.

| Method | Endpoint | Access | Description |
|--------|----------|--------|-------------|
| POST | `/auth/register` | Public | Create account |
| POST | `/auth/login` | Public | Log in, receive JWT |
| GET | `/articles` | Public | List approved articles |
| GET | `/articles/pending` | Editor | Articles awaiting review |
| POST | `/articles` | Journalist | Submit new article |
| PATCH | `/articles/:id/approve` | Editor | Approve or reject article |
| GET | `/articles/subscribed` | Reader | Personalised feed |
| GET | `/newsletters` | Public | List newsletters |
| POST | `/newsletters` | Journalist | Create newsletter |
| GET | `/publishers` | Public | List publishers |
| POST | `/subscriptions/publishers/:id` | Reader | Follow a publisher |
| POST | `/subscriptions/journalists/:id` | Reader | Follow a journalist |
| GET | `/stats` | Public | Platform-wide counts |

Full API reference is in [`planning/08-api-specification.md`](planning/08-api-specification.md).

## How to Run Tests

This project uses a manual testing approach documented in [`planning/09-test-plan.md`](planning/09-test-plan.md). Tests cover all three roles and the full article approval workflow.

### Manual testing with demo accounts

Start both servers (see Getting Started), then log in with the demo accounts to test each role:

**As an Editor (alice@pressroom.com):**
1. Log in and open the Dashboard — the pending articles queue should be visible
2. Click Approve or Reject on a pending article and confirm the status changes
3. Verify approved articles appear in the public articles feed

**As a Journalist (ben@pressroom.com or cara@pressroom.com):**
1. Log in and navigate to Write — submit a new article
2. Open the Dashboard and confirm the article shows a `pending` status badge
3. Create a newsletter and link articles to it

**As a Reader (dan@pressroom.com):**
1. Log in and follow a publisher and a journalist from their profile pages
2. Open the Dashboard and confirm the personalised feed shows articles from followed sources
3. Unfollow and confirm the feed updates

### API testing with curl

```bash
# Obtain a token
TOKEN=$(curl -s -X POST localhost:80/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"alice@pressroom.com","password":"password123"}' \
  | jq -r '.token')

# List pending articles (editor only)
curl -s localhost:80/api/articles/pending \
  -H "Authorization: Bearer $TOKEN" | jq .

# Approve an article (replace :id with a real article ID)
curl -s -X PATCH localhost:80/api/articles/:id/approve \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"approved": true}' | jq .
```

## Architecture

The API contract (`openapi.yaml`) is the single source of truth. Orval reads it and generates:
- **Zod schemas** (`lib/api-zod`) used by the Express server for request validation
- **React Query hooks** (`lib/api-client-react`) used by the frontend for data fetching

This means no fetch calls or validation schemas are ever written by hand — changes to the API flow automatically to both the server and the client.

Authentication is stateless JWT — tokens are stored in `localStorage` and auto-injected into every request via a custom fetch wrapper. The server enforces role-based access on every protected route using the `requireRole` middleware.
